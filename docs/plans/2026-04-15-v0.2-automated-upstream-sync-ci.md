# v0.2 — Automated Upstream Sync (CI) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Enhance the GitHub Actions sync workflow with rich PR descriptions, configurable frequency, webhook notifications, and CI validation gates.

**Architecture:** Enrich `SyncReport` with detailed diff data (added/removed/changed plugins per platform). The workflow gains `workflow_dispatch` inputs for frequency control, a notification step using webhook secrets, and a separate CI job that gates PR merges with type-check + tests.

**Tech Stack:** Bun, TypeScript, GitHub Actions, curl (webhooks)

- [ ] **Step 3: Implement formatSyncReportAsMarkdown**

  ```typescript
  // src/sync/report-formatter.ts
  import type { SyncReport, SyncReportEntry } from "./pipeline";

  export function formatSyncReportAsMarkdown(report: SyncReport): string {
    const lines: string[] = [];
    lines.push("## Sync Summary");
    lines.push("");
    lines.push(`**${report.updated} updated** out of ${report.total} total plugins.`);
    lines.push("");

    if (report.added.length === 0 && report.removed.length === 0 && report.changed.length === 0) {
      lines.push("No changes detected.");
      return lines.join("\n");
    }

    const renderSection = (title: string, entries: SyncReportEntry[]) => {
      if (entries.length === 0) return;
      lines.push(`### ${title} (${entries.length})`);
      lines.push("");
      for (const entry of entries) {
        lines.push(`- \`${entry.name}\` (${entry.platform})`);
      }
      lines.push("");
    };

    renderSection("Added", report.added);
    renderSection("Removed", report.removed);
    renderSection("Changed", report.changed);

    return lines.join("\n");
  }
  ```

### Task 6: Webhook Notification on PR Creation

**Files:**
- Modify: `.github/workflows/sync.yml`

- [ ] **Step 1: Add notification step after PR creation**

  ```yaml
  - name: Notify via webhook
    if: steps.changes.outputs.has_changes == 'true'
    env:
      SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
      DISCORD_WEBHOOK_URL: ${{ secrets.DISCORD_WEBHOOK_URL }}
    shell: bash
    run: |
      PR_URL=$(gh pr list --head "$BRANCH_NAME" --json url --jq '.[0].url')
      PAYLOAD='{"text":"🔄 Sync PR created: '"$PR_URL"'"}'

      if [[ -n "${SLACK_WEBHOOK_URL:-}" ]]; then
        curl -sfS -X POST -H 'Content-Type: application/json' \
          -d "$PAYLOAD" "$SLACK_WEBHOOK_URL" || echo "::warning::Slack notification failed"
      fi

      if [[ -n "${DISCORD_WEBHOOK_URL:-}" ]]; then
        DISCORD_PAYLOAD='{"content":"🔄 Sync PR created: '"$PR_URL"'"}'
        curl -sfS -X POST -H 'Content-Type: application/json' \
          -d "$DISCORD_PAYLOAD" "$DISCORD_WEBHOOK_URL" || echo "::warning::Discord notification failed"
      fi
  ```

  Key points:
  - Both webhooks are optional — the step only calls the ones with secrets configured.
  - Failures are non-fatal (`|| echo "::warning::..."`) so the sync itself isn't blocked.
  - `BRANCH_NAME` must be available (hoist it to job-level env or reference from prior step).

- [ ] **Step 2: Hoist BRANCH_NAME to job-level or extract PR URL from the create step**

  Cleanest approach: capture PR URL as a step output from the "Create commit and pull request" step, then reference in the notification step.

  ```yaml
  - name: Create commit and pull request
    id: create_pr
    # ... existing ...
    run: |
      # ... existing commands ...
      PR_URL=$(gh pr create ... 2>&1 | tail -1)
      echo "pr_url=$PR_URL" >> "$GITHUB_OUTPUT"

  - name: Notify via webhook
    if: steps.create_pr.outputs.pr_url != ''
    env:
      PR_URL: ${{ steps.create_pr.outputs.pr_url }}
      SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
      DISCORD_WEBHOOK_URL: ${{ secrets.DISCORD_WEBHOOK_URL }}
    shell: bash
    run: |
      if [[ -n "${SLACK_WEBHOOK_URL:-}" ]]; then
        curl -sfS -X POST -H 'Content-Type: application/json' \
          -d '{"text":"🔄 Sync PR created: '"$PR_URL"'"}' \
          "$SLACK_WEBHOOK_URL" || echo "::warning::Slack notification failed"
      fi
      if [[ -n "${DISCORD_WEBHOOK_URL:-}" ]]; then
        curl -sfS -X POST -H 'Content-Type: application/json' \
          -d '{"content":"🔄 Sync PR created: '"$PR_URL"'"}' \
  ```

## Verification Criteria

1. `bun test` passes with all new and existing tests green.
2. `bun run build` produces no type errors.
3. Workflow syntax is valid (`act` dry-run or manual `workflow_dispatch` trigger).
4. Sync PR body contains Markdown with added/removed/changed sections when plugins change.
5. Sync PR body says "No changes" when nothing changed (edge case: workflow should not create PR at all, but formatter handles it gracefully).
6. Webhook step is no-op when secrets are not configured (no failure).
7. CI workflow blocks PR merge when type-check or tests fail.
